Lumina Distributed Multimodal Semantic Search Engine

Feb 2026 — Mar 2026

Lumina is an end-to-end multimodal retrieval system that indexes 8,092 images and retrieves them from natural-language descriptions without manually authored tags.

Problem and objective

Conventional image search usually depends on filenames, captions, or manually maintained tags. That works when a collection is small and carefully curated, but it breaks down when thousands of images arrive without consistent metadata. A query such as “a tall building at sunset” may describe the content perfectly while sharing no literal words with the stored filename.

The objective of Lumina was to make visual content searchable by meaning. The system needed to encode images once, accept open-ended text at query time, retrieve the nearest visual matches, and expose the result through both an API and a usable interface.

System architecture

I separated Lumina into ingestion, storage, retrieval, and presentation concerns. This prevents the expensive image-encoding stage from being coupled to every search request.

  • Ingestion pipeline: loads JPEG and PNG files, encodes each image with clip-ViT-B-32, attaches file-path and timestamp metadata, and upserts points into Qdrant in batches.
  • Vector storage: Qdrant stores the 512-dimensional image vectors in the lumina_multimodal collection together with payload metadata.
  • Query service: FastAPI validates a text query and result limit, encodes the text through the same CLIP model, and asks Qdrant for the closest points.
  • Interface: Streamlit caches the model and database client, exposes a Top-K control, and renders results in a three-column grid with similarity scores.

Why CLIP works here

CLIP is trained so that related images and text occupy nearby regions of one embedding space. During ingestion, an image becomes a vector. During search, the user’s sentence becomes another vector of the same dimensionality. Qdrant can therefore rank images using vector similarity even though the query and stored object are different modalities.

This is the key design advantage: the system does not attempt to generate a caption for every image or perform keyword matching. It directly compares learned representations. New queries do not require re-indexing because the image vectors are stable.

Ingestion and hardware

The ingestion script detects Apple Metal Performance Shaders and otherwise falls back to CPU. Each valid image is opened with Pillow, encoded, and converted into a Qdrant point with a UUID. Uploads are chunked into batches of ten to remain comfortably below payload limits and to make failures easier to isolate.

The deployed search API follows the same device-selection logic. The health endpoint exposes the active device and the known index size, which provides a lightweight operational check before the interface sends user traffic.

Important implementation lesson

Model inference was not the only scaling concern. Batch size, vector payloads, file availability, secrets management, and cloud/local execution modes all affected whether the system behaved like a product rather than a notebook.

Failure modes and trade-offs

Semantic retrieval can return visually plausible but contextually surprising results when a query is abstract or the image collection is narrow. Similarity scores are relative ranking signals, not calibrated probabilities. The current cloud interface also stores file paths as payloads; when the original image file is not present in the cloud container, it can display the match and score but not the bitmap itself.

A production iteration would store durable object URLs instead of local paths, add evaluation queries with relevance judgments, capture latency percentiles, and support incremental ingestion rather than scanning a whole directory. The current public Streamlit source also contains fallback connection credentials; those values should be rotated and removed so deployment fails closed unless secrets are supplied by the environment.

Outcome and next steps

Lumina proved the full retrieval loop: encode, store, search, serve, and display. More importantly, it made the boundary between model quality and system quality concrete. A strong embedding model still needs dependable ingestion, payload design, health checks, secret handling, and a user interface that explains its results.
