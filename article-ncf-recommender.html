Neural Collaborative Filtering Movie Recommender

May 2025 — Jul 2025

An end-to-end movie recommendation product built around learned user and item embeddings, a non-linear rating model, a FastAPI inference layer, and a Streamlit client.

Problem and scope

A recommender must estimate preference from sparse observations. Each person rates only a small fraction of the catalogue, so a simple lookup table cannot generalize to unseen user–movie pairs. The goal was to learn compact representations of users and movies, use those representations to estimate ratings, and build a service that could rank unseen titles for a selected user.

The project uses the MovieLens small dataset: roughly 100,000 ratings from 600 users across approximately 9,000 movies. That scale is large enough to expose sparse-interaction problems while remaining practical for repeatable training and local experimentation.

Model architecture

The production repository implements an ItemBasedNCF model in PyTorch. It is important to describe the actual implementation precisely: user and movie IDs are mapped to separate 64-dimensional embeddings, concatenated into a 128-dimensional vector, and processed by a multilayer perceptron. The current serving model does not contain a separate GMF branch.

  • Learned representations: independent embedding tables capture latent factors for users and movies.
  • Non-linear interaction: dense layers of 128, 64, and 32 units use ReLU activations, batch normalization, and 0.3 dropout.
  • Bias correction: learned user, movie, and global bias terms capture systematic rating tendencies outside the interaction network.
  • Bounded output: the final estimate is clamped to the valid MovieLens rating range of 0.5 to 5.0.

Recommendation generation

At startup, the backend reconstructs stable mappings between the original MovieLens identifiers and contiguous embedding indices. It loads the trained .pth weights onto MPS, CUDA, or CPU, then switches the network to evaluation mode.

For a recommendation request, the handler first removes movies the user has already rated. It scores every remaining movie under torch.no_grad(), sorts the predictions from highest to lowest, and enriches the top results with title and genre metadata. This produces recommendations that are both ranked and immediately displayable.

API and interface

The FastAPI layer separates model concerns from the user interface. It exposes health status, movie and user lists, recommendation generation, and user rating history. Pydantic response models keep the contract explicit: each recommendation includes a movie identifier, title, genres, and predicted rating.

The Streamlit application consumes those endpoints rather than loading the model directly. Users can choose a profile, inspect rating history, request Top-K recommendations, and explore the predicted output without interacting with notebooks or Python objects.

Engineering trade-off

The current ranker scores unrated movies one at a time. This is clear and correct for a small catalogue, but batching candidate tensors would reduce Python overhead and become necessary at a larger scale.

Evaluation and limitations

Offline recommendation quality should be evaluated with a held-out interaction split using metrics aligned to the product goal—for example RMSE for rating estimation and Hit Rate or NDCG for Top-K ranking. The repository demonstrates the complete serving path, but the portfolio deliberately avoids inventing evaluation values that are not recorded in the source.

The system is also limited to known users and known movies. A new user has no learned embedding, producing a classic cold-start problem. Practical next steps would add popularity or content-based fallbacks, batch candidate scoring, persistent writes for new ratings, and model-version metadata in the health response.

What the project demonstrated

This project connected four usually separate tasks: preparing interaction data, training a latent model, serving deterministic inference, and designing an interface around the recommendations. The most valuable outcome was understanding that a recommender is not only a network—it is also identifier mapping, candidate filtering, metadata enrichment, API contracts, and a plan for cold-start behavior.
